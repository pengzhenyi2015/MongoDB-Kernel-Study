# 6. 索引
和其他数据库一样，索引在 MongoDB 的执行引擎中起着举足轻重的作用。    
下面，对于 MongoDB 中索引的类型，属性和使用经验准则进行讨论。
## 6.1 索引类型
### 普通 Btree索引
普通的 Btree 索引可以有 1个或者多个字段，定义索引时字段的顺序非常重要。因为在索引的使用上遵循最左前缀匹配原则。    
关于普通 Btree 索引的内核实现方式，可以参考前面 1.2 章节对于 KeyString 的讲述。

### Multikey 索引

MultiKey 索引主要用于数组字段，比如下面 3 条文档，由于 'a' 字段可能存在数组类型，因此创建的 `{a:1}` 索引是 MultiKey 索引：
```
mymongo:PRIMARY> db.coll7.find({})
{ "_id" : ObjectId("69343ba65413734a94573339"), "a" : 1 }
{ "_id" : ObjectId("69343be25413734a9457333a"), "a" : [ 1, 2, 3 ] }
{ "_id" : ObjectId("69343d475413734a9457333b"), "a" : [ 1, 2, 3, 1 ] }
```
在 MongoDB 内核中，会对数组中的每个字段创建索引，所以虽然上面的例子中只有 3 条文档，但是会对应 `1+3+3(去重)=7` 条索引项。      
在使用 `{a:{$gte:1, $lte:3}}` 进行查询时，会有 7 条对应的索引项满足条件，但是在[**去重**](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/index_scan.cpp#L196)之后会精准返回 3 条文档。      
其中去重机制为，在 IndexScan 流程中，对于 multikey 索引，会有一个专门的 [`std::unordered_set<recordId>`](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/index_scan.h#L170) 来进行去重。

MultiKey 索引的维护，也和普通的 btree 索引有些区别，主要有：     
1. 构建索引时，会对数组中的重复元素去重后，只对应一个索引项。    
2. 在修改索引 key 对应的数组时，会对修改前后的数组项先进行去重后，再求差值，决定哪些索引项需要增加，哪些需要删除。参考 setDifference 方法，里面会对两个 BSONObjSet（本质时 `std::set`）进行差值计算。    

为了方便理解 multiKey 索引的存储结构，假设对一个表创建了 {a: 1} 索引。并且包含如下 2 条数据：      
```
{_id: 1, a: [1,2,3,4,1]}  --> 对应 recordId1
{_id: 2, a: [1,2,[3,4]]}  --> 对应 recordId2
```
则第 1 条数据会对应 4 个索引项，对应的索引 key 是 `1，2，3 和4`， 虽然 key=1 在数组中出现了 2 次，但索引中不会创建重复的索引项。    
而第 2 条数据会对应 3 个索引项，对应的索引 key 是 `1, 2 和 [3,4]`， 需要注意的是 `[3,4]` 只对应一个索引项，里面的值不会继续拆分。

### Wildcard 索引
Wildcard 通配符索引，主要用于字段变化灵活的场景。     
比如有一个字段 "a", 可能对应了数字、字符串等基础类型，也可能对应了 Object 这种嵌套类型。如果希望对 "a" 字段下的所有嵌套字段也自动创建索引，则可以使用通配符索引。     
比如对 "a" 字段下的所有嵌套字段创建索引：
```
mymongo:PRIMARY> db.coll.createIndex({"a.$**":1})
```
则下面的查询都可以使用该通配符索引：
```
# 查 "a" 字段
db.coll.find({"a":1})
# 查 "a" 字段下的嵌套字段
db.coll.find({"a.b":2})
# 查嵌套字段的嵌套字段，依此类推
db.coll.find({"a.b.c":3})
```

更极端的，可以对每个字段都创建通配符索引(**谨慎使用，索引的维护开销会非常大**)：
```
mymongo:PRIMARY> db.coll.createIndex({"$**":1})
```

如果是普通索引，索引的 btree 结构中，是不会存储字段名字的。因为字段名字固定，只需要存储在元数据中即可。可以参考 1.2 章节关于 keystring 的详细讨论。        

但是对于通配符索引，由于字段名字不固定，因此索引的 btree 结构中需要存储字段名。具体来说，在创建通配符索引时，会遍历通配符字段下的所有字段（参考 [WildcardKeyGenerator::_traverseWildcard](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/index/wildcard_key_generator.cpp#L108)）,然后将 indexPath 和 indexKey 一起组成一个索引项（参考 [WildcardKeyGenerator::_addKey](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/index/wildcard_key_generator.cpp#L172)）。

### Geo 索引
MongoDB 的 Geo 索引分为 2d 和 2dsphere 两种。其中 2d 索引是基于 GeoHash 算法的平面几何，而 2dsphere 索引是基于 [Google S2 库](https://github.com/google/s2geometry/tree/master)的球面几何。      
具体区别列举如下：    
| 特性 | 2d 索引 | 2dsphere 索引 |
|------|---------|--------------|
| 数据模型 | 平面坐标（经纬度） | 球面坐标（经纬度） |
| 距离计算 | 欧几里得距离 | 圆弧距离 |
| 适用范围 | 小范围数据，如游戏地图、平面图 | 全球范围数据，如真实地理位置 |
| 支持的地理类型 | 点 | 点、线、多边形、几何体集合 |
| 精度 | 受哈希精度限制（默认 60cm, 最多到 1cm 左右） | 可配置的 S2 单元格精度 |
| 性能 | 索引构建和查询更快 | 考虑曲率，计算更精确但稍慢 |

#### 2d 索引

**基本思路**    

2d 索引将地球进行平面展开，然后对经度和维度分别进行编码。    
编码的大致思路是，经度或者维度越小，则编码为 0，反之，编码为1.     
通过不断递归上述流程，将地球平面划分为非常多的小方格。     
举例如下：
1. 通过 2 个 bit， 可以把地球分为 4 部分，其中第 1 个 bit 是经度，第 2 个 bit 是维度。左下角的经纬度最小，因此为 00， 右上角的经纬度最大，因此为 11.    
2. 如果继续递归，再多加 2 bit， 则可以将地球分隔为 `4*4 = 16` 份。
3. 依此类推，可以将地球划分为 `(2^26) * (2^26) = (2^52)` 个小方格。

可以发现上述分割算法具有如下特点：
1. 经维度**交叉编码**，而不是简单创建一个 经度+维度 的复合索引。
2. 一个父平面可以划分为 4 个子平面，这 4 个**子平面的前缀相同**。
3. 通过**前缀遍历**可以扫描某个平面的所有子平面， 前缀越短，范围越大。比如按照 `11` 前缀遍历，可以扫描到地球右上角的所有子平面。    

>为什么使用经纬度交叉编码，而不是简单的 经度+维度复合索引？    
>因为复合索引中，需要满足最左匹配原则，也就是说，不同字段的重要性是不一样的。而 geo 场景中，经度和维度的重要性一样。    
>创建 geo 索引的目标，是把距离相近的点聚集在一起，便于快速检索。复合索引显然不能满足这个需求。        

更多 GeoHash 的理论知识，可以知乎文章 [Geohash编码计算方法](https://zhuanlan.zhihu.com/p/606033199)。

**编码流程**     

经纬度的交叉编码算法流程为：    
1. 将经纬度分别进行 `+180` 的平移，避免负数计算。平移后，经度的范围从 `[-180, 180)` 变为 `[0,360)`, 维度的范围从 `[-90, 90)` 变为 `[0,180)`.    
2. 乘以一个系数 (11930464), 将度数 `[0,360)` 从映射到 `[0, uint32::max)` 的空间。此时经纬度分别用 uint32 表示。
3. 将经纬度进行交叉编码，将每个 uint32 的值的前 26 位抽取出来，拼接成一个 52 bit 的 key。通过 uint64 存储（前 52 位存储交叉编码的经纬度，后 12 位保留）。    

第 1、2 步的流程可以参考 [GeoHashConverter::convertToDoubleHashScale](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/geo/hash.cpp#L870):
```
// 将经度和维度转换为 unsigned 整型的过程 GeoHashConverter::convertToHashScale ->
//                                            GeoHashConverter::convertToDoubleHashScale
//  min = -180, max = 180, scaling = 11930464.711111112, _error = 7.5865165509926415e-06
double GeoHashConverter::convertToDoubleHashScale(double in) const {
    verify(in <= _params.max && in >= _params.min);

    if (in == _params.max) {
        // prevent aliasing with _min by moving inside the "box"
        // makes 180 == 179.999 (roughly)
        in -= _error / 2;
    }

    in -= _params.min;  // 加 180，转成， [0,360) 之间的浮点数
    verify(in >= 0);
    return in * _params.scaling;  // 乘以 11930464cm，从弧度映射成 360 * scaling == uint32 最大值（4,294,967,296）
                                 //， 更详细的可以参考 GeoHashConverter::parseParameters
}
```

第 3 步的流程可以参考 [GeoHash::GeoHash](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/geo/hash.cpp#L155)：
```
// 这里的 x, y 是通过 转换而来的，bits=26. 即经度和纬度各 26 bit
// 具体的执行流程，可以理解为，将 uint32 的经纬度信息Hash到 uin64 的 _hash 中，而是每个维度
// 只取高位的 26 bit 信息（也就是说经度从 1cm 左右下降到 60 cm）
// Hash 的方式，是从高 bit 位开始，轮流映射经度和维度信息。
GeoHash::GeoHash(unsigned x, unsigned y, unsigned bits) {
    verify(bits <= 32);
    _hash = 0;
    _bits = bits;
    for (unsigned i = 0; i < bits; i++) {
        if (isBitSet(x, i))
            _hash |= mask64For(i * 2);
        if (isBitSet(y, i))
            _hash |= mask64For((i * 2) + 1);
    }
}
```

**为什么选择 26 bit 作为交叉编码的精度？而不是用满 32 bit?**    
如果使用 32 bit, 那么精度可以从 60cm 提升到 1cm 左右，但是计算开销会更大（比如下文要提到的起始步长计算逻辑）。而且大多数情况下60cm 的精度已经足够。    
      
MongoDB 也提供了参数，可以在创建 2d 索引时，自己制定精度：
```
db.<collection>.createIndex( {<location field> : "<index type>"} ,
                             { bits : <bit precision> } )
```
官方文档也说明了使用 26 bit 精度的原因：    
>By default, a 2d index on legacy coordinate pairs uses 26 bits of precision, which is roughly equivalent to 2 feet or 60 centimeters of precision using the default range of -180 to 180. Precision is measured by the size in bits of the geohash values used to store location data. You can configure geospatial indexes with up to 32 bits of precision.

>Index precision does not affect query accuracy. The actual grid coordinates are always used in the final query processing. Advantages to lower precision are a lower processing overhead for insert operations and use of less space. An advantage to higher precision is that queries scan smaller portions of the index to return results.

**检索流程**

通过上文描述的 GeoHash 编码之后，已经能够通过 btree 将距离相近的点聚集在一起了。       

以检索某个中心点周围 10km 的所有点，并从近到远排序的典型业务场景为例。 使用 2d 索引的检索策略为：
1. 对中心点附近的数据密度进行分析，确定扫描的起始步长。
2. 根据中心点，和起始步长画圆，求解圆形和 GeoHash 索引的交集，确定需要扫描的“方格”数组，这些“方格”可以使用 GeoHash 表示，因此非常利于使用 GeoHash btree 索引进行范围检索。
3. 根据检索出的结果集大小，确定下一轮的步长。
4. 根据本轮的步长，和下一轮的步长，制定出下一轮要扫描的 “环形” 区域，并计算“环形”区域和 GeoHash 相交的"方格"数组。
5. 重复上述步骤，直到扫描完所有符合条件的结果集。

上述流程中，有几个关键点，非常值得分析和学习：     
1. 如何确定起始步长？        
   确定起始步长非常关键，如果起始步长过大，可能第一轮扫描的点过多，甚至远超过需要的结果集大小。这样很多资源会浪费在无效的扫描和排序操作上。    
   如果起始步长太小，则会执行很多轮，同样效率很低。甚至会出现前 N 轮扫描不到任何有效结果的情况。

   MongoDB 对此问题的解法为：从中心点开始，快速（指数级）扩大扫描范围，只要成功找到第 1 个点，则可以估算起始步长了。     
   示意图如下：    

   <p align="center">
	<img width="796" height="750" alt="image" src="https://github.com/user-attachments/assets/ae507dc1-9b5c-4142-ae52-5fcd1c127d6b" />
   </p>

   假设图中使用对角线标识的是起步的中心点位置（或者上一轮搜索完成的平面），则使用下述算法确定下一轮要扫描的平面：    
   a. 将 `level--` (即经纬度的前缀分别减 1)，得到当前子平面的父平面， 将父平面加入到下一轮扫描的候选集中。    
   b. 根据子平面在父平面中的方位（可能左上、左下、右上、右下， 本例是左上） ，首先添加 X 轴（经度）的邻居，本例中是左侧的邻居。    
   c. 根据子平面在父平面中的方位，添加 Y 轴（纬度）的邻居。本例中是上方的邻居。    
   d. 根据子平面在父平面中的方位，添加对角线邻居。本例中是左上方的对角线邻居。    
   e. 将上述 4 个平面排序后，作为索引扫描条件，进行下一轮扫描。    
   上述算法可以参考 [GeoHash::appendVertexNeighbors](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/geo/hash.cpp#L565)。每轮扫描的范围，呈指数级增长。     
   如果成功扫描到一个结果，则停止扫描，并进行距离估算，然后确定起始步长。 参考 [GeoNear2DStage::DensityEstimator::work](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/geo_near.cpp#L353) 中如何求解 estimatedDistance, 以及 [GeoNear2DStage::initialize](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/geo_near.cpp#L428) 中如何使用 estimatedDistance 设置初始步长（乘以3）.
2. 如何动态调整步长？    
   根据本轮扫描的结果集大小，确定下一轮的步长。如果本轮扫描的结果集太小，则比较激进的增加步长，反之则减少步长增长幅度。    
   参考 [GeoNear2DStage::nextInterval](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/geo_near.cpp#L594).
   ```
	// 设置下一个间隔
    if (!_specificStats.intervalStats.empty()) {
        const IntervalStats& lastIntervalStats = _specificStats.intervalStats.back();

        // 根据上一次返回的结果数量调整步长
        if (lastIntervalStats.numResultsReturned < 300)
            _boundsIncrement *= 2;  // 结果太少，加倍步长
        else if (lastIntervalStats.numResultsReturned > 600)
            _boundsIncrement /= 2;  // 结果太多，减半步长
    }
   ```
3. 如何避免重复扫描？    
   可能会有方块同时出现在不同轮次的扫描中，如下图黄色标注的方块所示：     

   <p align="center">
   <img width="375" height="371" alt="image" src="https://github.com/user-attachments/assets/e32d19b8-45a5-42b7-a57d-327383cce433" />
   </p>

   为了避免重复扫描，每轮扫描的方块会被[记住](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/geo_near.cpp#L709)，在后续的扫描中会被[排除](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/geo_near.cpp#L703)。
   另外，在文档级别，也会使用一个 `unordered_map` 记录返回的文档 id， 进行[去重](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/near.cpp#L189)。

3. 如何进行排序和过滤？      
   这个扫描过程分为： Initializing（确定初始步长）-> Buffering(确定扫描区域、扫描、缓存并排序) -> Advancing（返回筛选后的结果）-> Finished 共 4 个阶段。每轮扫描都有 Buffering 和 Advancing，所以这 2 个阶段可能会执行多次。       
        
   排序操作在 Buffering 阶段，具体来说：      
   使用了一个 [NearStage::SearchResult](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/near.cpp#L122) 结构的优先队列。每次缓存结果集时，会传入 memberId 和 distance 到[优先队列](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/near.cpp#L211)，优先队列会使用 distance 进行升序排列。       
   而过滤操作处于每一轮的 Advancing 阶段：      
   如果结果的距离小于本轮的 minDistance，则直接丢弃。如果超过了本轮的 maxDistance，则先不返回，继续保留在缓存中，等待下一轮处理。    
   参考 [NearStage::advanceNext](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/near.cpp#L221).

#### 2dsphere 索引

**编码流程**      

S2 库采用将球面坐标，映射到一个 3 维立方体中，然后进行校准后，采用 Hilbert 曲线进行编码。      
关于这部分的理论知识，可以参考知乎文章 [Google S2空间索引概述](https://zhuanlan.zhihu.com/p/611457995) 或者[S2官方 PPT](https://docs.google.com/presentation/d/1Hl4KapfAENAOf4gv-pSngKwvS_jwNVHRPZTTDzXXn6Q/view?slide=id.i0#slide=id.i0).

下面通过对一部分 S2官方 PPT 进行分析，尝试快速理解 S2 索引的编码流程(在 S2 代码库中实现，不是 MongoDB 内核代码，因此不做代码级分析)：       

1. 地球表面的点可以通过一个 3 维坐标系表示，其中 x,y,z 的取值均为 [-1,1].    
   如果将经纬度投影到 3 维坐标系的过程定义为 p, 则 p = `(lat,lng) => (x,y,z)`.

   </p>

2. 在 3 维坐标系中，沿着地球表面生成一个立方体。则从球心出发，可以将 3 维坐标系中的点映射到立方体的一个面上。    
   这个映射过程为 `(x,y,z) => (face, u, v)`， 其中 face 为立方体其中一个表面的编号，u,v 为该平面的坐标。      
   但是这个映射存在一个问题：越靠近平面的边缘处，失真越严重，既面积会被放大。

	<p align="center">
	<img width="533" height="400" alt="Geometry on the Sphere_ Google&#39;s S2 Library (1)" src="https://github.com/user-attachments/assets/d4465493-da98-47c5-b477-348fa5f316d6" />
	</p>

3. 为了解决平面边缘失真的问题， S2 增加了一个非线性映射：`(face,u,v) => (face,s,t)`. 其中 u,v 的值域为 `[-1,1]`, s,t 的值域为 `[0,1]`.

   <p align="center">
   <img width="533" height="400" alt="Geometry on the Sphere_ Google&#39;s S2 Library (2)" src="https://github.com/user-attachments/assets/49a4c826-fe41-4b09-8f05-0522faaadccf" />
   </p>

4. 官方分析了多种映射算法，最后选择了 Quadratic：    
   a. Linear: 速度快，但是效果不好。比如 `s = 0.5 * (u+1)` 和 `t = 0.5 * (v+1)` 就能实现 `[-1,1]` 到 `[0,1]` 的映射，但是解决不了任何失真的问题。    
   b. Tangent: 效果好，但是速度慢；    
   c. Quadratic: 速度快，而且效果也不错；

   <p align="center">
   <img width="533" height="400" alt="Geometry on the Sphere_ Google&#39;s S2 Library (3)" src="https://github.com/user-attachments/assets/b95ac391-1486-40fb-ae91-23b2820e6e79" />
   </p>

5. 经过了上述转换后，再将平面进行切分，变成 `2^30 * 2^30` 个小平面（精度在 cm 级别）。     
   将这个转换定义为 `(face, s, t) => (face, i, j)`， 其中 i, j 的值域为 `[0, 2^30 -1]`.

	<p align="center">
	<img width="533" height="400" alt="Geometry on the Sphere_ Google&#39;s S2 Library (4)" src="https://github.com/user-attachments/assets/26f54341-4bb1-41d8-bcf7-916e48e82385" />
	</p>


6. 使用 Hilbert 曲线，将 `(face, i, j)` 编码为一个 64 位的整数，称为 S2CellId。空间位置相邻的点，S2CellId 也相邻。         

	<p align="center">
	<img width="533" height="400" alt="Geometry on the Sphere_ Google&#39;s S2 Library (5)" src="https://github.com/user-attachments/assets/378317c4-2261-4a90-b98c-81a8f6eac11c" />
	</p>

	<img width="533" height="400" alt="Geometry on the Sphere_ Google&#39;s S2 Library (6)" src="https://github.com/user-attachments/assets/e0c941f3-f0d5-4a46-977f-ee103ae5318d" />
	</p>

8. S2CellId 非常适合[作为 btree 索引的 key](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/index/s2_common.cpp#L74). 而且和 GeoHash 索引一样， level 每增加 1， 索引的范围就会缩小 4 倍。

**检索流程**       

通过上述 S2 库中的 S2CellId 编码之后，已经可以将距离相近的点通过 btree 索引组织在一起了。          
整体执行计划和 2d 索引相同，也是包括 Initializing -> Buffering -> Advancing -> Finished 4 个阶段。如果有多轮检索，则 Buffering -> Advancing 可能持续多轮。      

从代码可以看到，主要的执行流程在 [NearStage](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/near.h#L90) 中，其中 initNext(), bufferNext(), AdvanceNext() 方法定义了上述阶段的执行流程。 而 GeoNear2DStage 和 GeoNear2DSphereStage 都是继承此类。    
不同的点在于， [GeoNear2DStage](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/geo_near.h#L67) 和 [GeoNear2DSphereStage](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/exec/geo_near.h#L107) 各自 override 了继承的 nextInterval(), computeDistance() 和 initialize() 方法。这些方式的实现不同，则体现了平面索引（GeoHash）和球面索引（S2）的区别。

#### 2d 还是 2dsphere 索引？

通过前面的分析，我们不难看出：     
- 2d 索引的转换更简单，理论上速度更快；
- 2dsphere 索引转换更复杂，理论上速度更慢，但是由于对映射进行了矫正，因此会更加精确。

为了比较形象地认识到这一点，我们分别使用上述 2 种索引，对 2 种极端场景进行 `$geoNear` 查询:    
- 经度回绕：-179 和 179 这两个经度的数值相差很大（几乎是一个地球的周长），但是实际上在球面距离上只有 2 度；
- 南北极问题： 在极地，即使移动一小段距离，经纬度也会剧烈波动。

测试结果如下（注意： 2d 索引的 maxDistance 单位是角度，而 2dsphere 索引的 maxDistance 单位是米）：

### Hashed 索引
MongoDB 的 Hashed 索引，会用 btree 结构对 key 的 hash 值进行索引。     
和其他很多数据库不一样（比如 PostgreSQL 的 Hash 索引），**MongoDB 并不会使用 hash 表存储索引**，而是用 btree.

对于一个很长的字段，比如一个 1000 个字符的字符串，如果直接使用该字段作为索引 key，则索引非常大。如果使用 Hash 索引，由于使用的是 8 字节的 Hash 值创建索引，因此索引大小很大大降低。关于 Hash 索引的空间优势，可以参考 6.2 章节的讨论。    

## 6.2 索引属性
### Collation(Case-Insensitive)
MongoDB 内核使用 ICU(International-Component-for-Unicode) 库提供了对字符的国际化支持。    
比如常见的使用方式，就是在查询的时候忽略大小写和音标等。    

一个典型的例子如下：
```
插入数据：
db.coll3.insertMany([{a:"apple"}, {a:"Apple"}, {a:"APPLE"}, {a:"banana"}])

创建索引：
db.coll3.createIndex({a:1},{collation:{locale:"en", strength:2}})

读取数据：
db.coll3.find({a:"apple"}).collation({locale:'en', strength:2})
能够查询 "apple","Apple" 和 "APPLE".
```

如果定义表时指定了 [collation](https://www.mongodb.com/docs/manual/reference/collation/#std-label-collation-document-fields)，则该表的索引和查询默认会继承该 collation.

collation 使用标准的 icu 库进行索引 key 的转换， 参考 [CollatorInterfaceICU::getComparisonKey](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/query/collation/collator_interface_icu.cpp#L75).


### Hidden
MongoDB 高版本（建议 6.0 以上，本文测试用 8.0.9）支持隐藏索引。    
隐藏索引可以用于评估这个索引如果被删除，是否会有严重的负面影响。    

需要注意的是，索引隐藏后只是对[执行计划不可见](https://github.com/mongodb/mongo/blob/r8.0.9/src/mongo/db/query/query_planner_params.cpp#L197)（包括自动选择执行计划，或者 hint 强制指定执行计划），但是被隐藏的索引在后台还是在维护中。因此，隐藏的索引还是会有维护成本，占用空间，而且依然需要满足索引的相关约束，如 Unique 约束，TTL 执行动作等。    

用户可以在 createIndex 时指定 {hidden: true} 直接创建一个隐藏状态的索引。    
也可以使用 hideIndex 和 unhideIndex 来隐藏和暴漏索引，由于不涉及真正的索引删除和创建动作，因此这个过程非常快。    


### Partial
Partial 索引，即只对满足特定条件的文档创建索引。    
例如，只对 age > 18 的用户创建索引：   
```
db.coll3.createIndex({age:1}, {partialFilterExpression:{age:{$gt:18}}})
```
通过这种方式可以减少索引的维护成本，降低索引的大小。    

MongoDB 在生成执行计划时，会检查查询条件是否为 partialFilterExpression 的[子集](https://github.com/mongodb/mongo/blob/r4.2.25/src/mongo/db/query/plan_cache_indexability.cpp#L54)，如果是，则可以使用该索引。    
比如上面的例子，如果查询条件时 `{age:{$gt: 20}}` 则可以使用索引，如果是 `{age:{$gt: 15}}` 则不可以使用。    	

### Sparse
Sparse 稀疏索引，即只对字段存在的文档创建索引。
也可以理解为 partial 索引使用 { key： {$exists: true} } 条件的特化版本。

### TTL
TTL 索引用于自动淘汰过期文档。     
使用方式、底层原理和索引注意事项，参考 5.2 章节的讨论。        

### Unique
用户可以创建 unique 索引进行字段的唯一性约束，当尝试插入相同 key 的文档时，会报 "duplicate key" 错误。     

关于 MongoDB 进行 unique 检查的底层原理，可以参考 1.2 章节中的相关讨论。    

很多用户习惯使用 upsert 操作来处理 "duplicate key" 报错。但是如果没有创建唯一索引，可能由于并发操作导致出现重复 key 的情况。    
例如：
```
# session1 执行 upsert
mymongo:PRIMARY> s = db.getMongo().startSession()
session { "id" : UUID("a7bc3ed7-cb8c-4db1-950b-2f82d64ec7fc") }
mymongo:PRIMARY> s.startTransaction()
mymongo:PRIMARY> coll5=s.getDatabase("db1").coll5
db1.coll5
mymongo:PRIMARY> coll5.update({a:3},{$set:{b:1}},{upsert:true})
WriteResult({
        "nMatched" : 0,
        "nUpserted" : 1,
        "nModified" : 0,
        "_id" : ObjectId("692c3f349b543feb807d310b")
})

# session2 执行同样的 upsert
mymongo:PRIMARY> s2 = db.getMongo().startSession()
session { "id" : UUID("3aedec76-13f7-4482-9ab1-2ca7fa64bf52") }
mymongo:PRIMARY> s2.startTransaction()
mymongo:PRIMARY> coll5=s2.getDatabase("db1").coll5
db1.coll5
mymongo:PRIMARY> coll5.update({a:3},{$set:{b:1}},{upsert:true})
WriteResult({
        "nMatched" : 0,
        "nUpserted" : 1,
        "nModified" : 0,
        "_id" : ObjectId("692c3f329b543feb807d3107")
})

# session1 提交事务
mymongo:PRIMARY> s.commitTransaction()
# session2 提交事务
mymongo:PRIMARY> s2.commitTransaction()

# 查表，发现有两个相同的 key ！！
mymongo:PRIMARY> db.coll5.find({a:3})
{ "_id" : ObjectId("692c3f329b543feb807d3107"), "a" : 3, "b" : 1 }
{ "_id" : ObjectId("692c3f349b543feb807d310b"), "a" : 3, "b" : 1 }
```
究其原因，是数据库中没有唯一索引，所以 upsert 能不能预期执行，全凭运气（是否有相同 key 并发 upsert）。    
**所以，如果要使用 upsert 操作，切记看看有没有创建 unique 索引。**

相对而言，PostgreSQL 在执行 `insert into ... on conflict do update` 时，会先检查是否有唯一索引，如果没有则会报错。个人认为这种直接报错的行为，要更加严谨一点。    
例如：
```
postgres=# create table t1 (c1 int, c2 int);
CREATE TABLE
postgres=# insert into t1 values (1,1) on conflict(c1,c2) do update set c2=3;
ERROR:  there is no unique or exclusion constraint matching the ON CONFLICT specification
```

# 附录 3： 2d 索引和 2dsphere 索引在经度回绕和南北极极值点的差异对比脚本
```
/**
 * geonear_longitude_wrap_and_polar_tests.js
 * 
 * This test covers two specific scenarios for geonear queries:
 * 1. Cross-longitude searches (from -179 to 179 at the same latitude)
 * 2. Polar region searches near the North and South poles
 * 
 * Both 2d and 2dsphere indexes are tested for these scenarios.
 */

(function() {
    "use strict";

    const conn = MongoRunner.runMongod();
    assert.neq(null, conn, "mongod failed to start.");
    const db = conn.getDB("test");
    const collName = "geonear_test";
    let coll;

    // Helper function to drop and create collection
    function setupCollection() {
        coll = db.getCollection(collName);
        coll.drop();
        return coll;
    }

    // Test 1: Cross-longitude search (from -179 to 179 at same latitude)
    jsTest.log("Testing cross-longitude search with 2d index");

    coll = setupCollection();
    
    // Insert points around the date line
    const crossLongitudePoints = [
        { _id: 1, loc: [-179, 0] },   // Close to -180
        { _id: 2, loc: [178, 0] },    // Close to +180
        { _id: 3, loc: [179, 0] },    // Close to +180
        { _id: 4, loc: [-178, 0] },   // Close to -180
        { _id: 5, loc: [0, 0] },      // Far from date line
        { _id: 6, loc: [-179.5, 0.5] }, // Near date line, slightly north
        { _id: 7, loc: [179.5, -0.5] } // Near date line, slightly south
    ];
    
    assert.commandWorked(coll.insert(crossLongitudePoints));
    assert.commandWorked(coll.createIndex({ loc: "2d" }));
    
    // Query from point near -179 with large enough radius to cross the date line
    let result = coll.find({
        loc: {
            $near: [-179, 0],
            $maxDistance: 3  // This should span across the date line
        }
    }).toArray();
    
    print("Cross-longitude 2d index results (near [-179, 0]):");
    result.forEach(doc => printjson(doc));
    // { "_id" : 1, "loc" : [ -179, 0 ] }
    // { "_id" : 6, "loc" : [ -179.5, 0.5 ] }
    // { "_id" : 4, "loc" : [ -178, 0 ] }
    
    // Test with $geoNear aggregation with 2d index
    let geoNearResult = coll.aggregate([
        {
            $geoNear: {
                near: [-179, 0],
                distanceField: "dist",
                spherical: false,  // 2d index, so not spherical
                maxDistance: 3,
                includeLocs: "loc"
            }
        }
    ]).toArray();
    print("$geoNear 2d index results (near [-179, 0]):");
    geoNearResult.forEach(doc => printjson(doc));
    // { "_id" : 1, "loc" : [ -179, 0 ], "dist" : 0 }
    // { "_id" : 6, "loc" : [ -179.5, 0.5 ], "dist" : 0.7071067811865476 }
    // { "_id" : 4, "loc" : [ -178, 0 ], "dist" : 1 }

    // Test with 2dsphere index
    jsTest.log("Testing cross-longitude search with 2dsphere index");
    
    coll = setupCollection();
    assert.commandWorked(coll.insert(crossLongitudePoints));
    assert.commandWorked(coll.createIndex({ loc: "2dsphere" }));
    
    // Query with 2dsphere index
    result = coll.find({
        loc: {
            $nearSphere: {
                $geometry: { type: "Point", coordinates: [-179, 0] },
                $maxDistance: 333000  // ~3 degrees in meters
            }
        }
    }).toArray();
    
    print("Cross-longitude 2dsphere index results (near [-179, 0]):");
    result.forEach(doc => printjson(doc));
    // { "_id" : 1, "loc" : [ -179, 0 ] }
    // { "_id" : 6, "loc" : [ -179.5, 0.5 ] }
    // { "_id" : 4, "loc" : [ -178, 0 ] }
    // { "_id" : 7, "loc" : [ 179.5, -0.5 ] }
    // { "_id" : 3, "loc" : [ 179, 0 ] }
    
    // Test with $geoNear aggregation using 2dsphere
    geoNearResult = coll.aggregate([
        {
            $geoNear: {
```
