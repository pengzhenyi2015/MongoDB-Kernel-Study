# 1. 导语

审计日志用于记录用户的关键操作，是一种事后追溯的重要手段。    

MongoDB 企业版提供了审计功能，对于常见的 MongoDB 操作提供了丰富的审计日志记录。对提供了丰富的过滤规则，以及输出格式和存放位置。     
具体可以参考[官方文档](https://www.mongodb.com/docs/manual/core/auditing/)。

# 2. Percona MongoDB Server 的审计日志
Percona 的审计功能是 MongoDB Enterprise 审计的开源替代方案，在核心功能（配置参数、事件格式、过滤器、日志目标）上完全兼容，但在较新的高级特性（OCSF 格式、运行时过滤器管理）上存在差距，同时提供了一些自己的独有扩展（auditGetOptions 命令、directAuthMutation 事件等）。     

以 percona-server-mongodb-psmdb-8.0.21-9 版本为例，其审计结构的设计如下：
```
logv2::AuditLog (旋转接口)
    └── WritableAuditLog (抽象: 过滤 + 旋转 + flush + fsync)
        ├── FileAuditLog (抽象: 文件写入 + 互斥锁 + 重试)
        │   ├── JSONAuditLog  (JSON格式文件)
        │   └── BSONAuditLog  (BSON格式文件)
        ├── ConsoleAuditLog (控制台输出)
        ├── SyslogAuditLog  (syslog输出)
        └── VoidAuditLog    (debug构建时验证, 不实际写入)

```

写入审计日志的核心结构为  FileAuditLog，在写入日志时，会获取锁，然后追加到专属的内存缓冲区中。后台的刷盘线程，也需要或者这个锁，将内存缓冲区中的日志写入文件。    

参考 FileAuditLog::appendMatched 和 FileAuditLog::flush 方法的实现：
```
    void appendMatched(const BSONObj& obj, const bool affects_durable_state) override {
        boost::scoped_ptr<AuditLogFormatAdapter> adapter(createAdapter(obj));

        // mongo::File does not have an "atomic append" operation.
        // As such, with a rwlock we are vulnerable to a race
        // where we get the length of the file, then try to pwrite
        // at that offset.  If another write beats us to pwrite,
        // we'll overwrite that audit data when our write goes
        // through.
        //
        // Somewhere, we need a mutex around grabbing the file
        // offset and trying to write to it (even if this were in
        // the kernel, the synchronization is still there).  This
        // is a good enough place as any.
        //
        // We don't need the mutex around fsync, except to protect against concurrent
        // logRotate destroying our pointer.  Welp.
        stdx::lock_guard<SimpleMutex> lck(_mutex);   // 加锁，写内存缓冲区

        _dirty = true;
        if (affects_durable_state)
            _fsync_pending = true;

        invariant(_membuf.write(adapter->data(), adapter->size()));
    }


    void flush() override {
        stdx::lock_guard<SimpleMutex> lck(_mutex);   // 加锁，执行 flush

        if (_dirty) {
            flush_inlock();
            _dirty = false;
        }
    }

    void fsync() override {
        stdx::lock_guard<SimpleMutex> lck(_mutex);   // 加锁，执行 fsync

        if (_fsync_pending) {
            if (_dirty) {
                flush_inlock();
                _dirty = false;
            }

            fsync_inlock();
            _fsync_pending = false;
        }
    }

```

对于需要大并发、大量生成审计日志的场景，**这样的日志写入方式，显然会成为性能瓶颈**。

# 3. 一种高性能的审计日志方法

在高并发场景下开启全量审计日志后，数据库的整体吞吐会出现比较明显的下滑。     
在进行了性能分析以及相关开源代码的分析之后发现，主要存在 2 大性能瓶颈：    
1. IO 效率比较低。以某知名开源社区的实现为例，在生成一个审计日志之后，需要先拿互斥锁，然后写入到一个内存 buffer 中。后台线程每隔一段时间，将buffer 中的内容写入到磁盘文件中。这里存在 2 个明显的问题：
a. 多个线程都要抢锁，然后对 buffer 执行内存拷贝动作，如果有几千个并发线程，那么可能造成阻塞；
b. buffer的生产者（用户线程写审计信息）和消费者（后台线程获取 buffer 内容刷到文件中）需要抢同一个锁，可能相互阻塞。
2. BSON 和 JSON 的类型操作耗费性能。由于 BSON 格式可以快速地转换成 JSON 等类型，而且能够非常方便地进行规则匹配（比如指定某些来源 IP 的某个表打印审计日志），因此，一些主流的实现方式为先生成 BSON 格式的审计日志，然后再进行规则过滤，并转换成最终的格式进行输出。此时会存在 2 个明显的问题：
a. BSON 日志的生成，以及到 JSON 格式的转换比较耗费性能；
b. 生成的 BSON 审计日志，可能在规则匹配的时候发现是不需要记录的，此时也会浪费资源；

因此，需要在架构设计时，针对 IO 效率，以及 BSON/JSON 的类型操作进行了针对性地优化。    
## 异步批量刷机制
为了提升磁盘的 IO 效率，我们也引入了内存 buffer 进行异步批量刷。但是不同的点是：
1. 采用了双 buffer 机制，将生产者（用户线程写审计信息）和消费者（后台线程获取 buffer 内容刷到文件中）的流程分离，避免相互阻塞。为了保证安全性，还引入了一个读写锁，写 buffer 的流程在共享锁的保护下支持并发写入，而刷 buffer 的流程在独占锁的保护下完成双 buffer 的切换。由于buffer 切换的本质只是指针的赋值，因此占用独占锁的时间非常短暂。由于写 buffer 和刷 buffer 对应了不同的 buffer空间，因此将 buffer 中的数据写入到磁盘的 IO 操作，则可以在锁外安全地执行。
2. 尽量降低锁的范围，减少互斥。为了实现并发安全地写 buffer，我们也为 buffer 引入了一个 mutex。但是区别在于每次写 buffer 的流程在拿到锁之后，只需要获取可写的 buffer 空间即可(可写的起始位置，和长度)，将审计信息写入到 buffer 的具体执行流程是在锁外完成的。

后台刷盘流程有 2 种触发机制：100ms 定时唤醒，以及 buffer 使用率达到 90% 后主动唤醒。    

采用异步方式虽然能够得到性能提升，但是也存在一定的审计日志丢失风险。    
为了解决这个问题，我们也实现了同步刷审计日志的机制（不写内存buffer，而是直接拿到 audit.log 的文件锁后写文件），能解决以下问题：    
1. 充分保证审计日志的安全性。不过这种方式下会牺牲性能，用户需要根据自身的实际业务场景进行选择。
2. 在异步方式的极端场景下，如果磁盘性能太差，但是生成审计日志的速度又很快，可能导致 buffer 中的数据累计太多占用太多内存。在我们的设计中，buffer 会有最大大小的限制，累计的数据量超过限制之后，后续的审计日志也会切换到同步模式。直到buffer 可用之后再自动切换回异步模式。通过这种方式，可以实现性能、内存使用和审计信息安全 3 者之间的平衡。

## 定制化的审计日志生成方式
采用 BSON 方式来生成审计日志会比较简单，并且很容易实现规则过滤、类型转换（转JSON）等功能。但是我们再进行性能分析之后发现，BSON 的构造、匹配、类型转换等操作比较耗费 CPU，特别是在需要生成大量审计日志的场景，性能瓶颈凸显。     
针对这个问题，我们进行了如下优化：    
1. 将规则匹配的时机提前到审计日志生成之前，消除对 BSON 匹配操作的依赖，避免生成无效的审计日志。常用的匹配规则有库表名、IP、用户名、操作类型等，这些信息都是能够在生成审计日志之前获取到的。    
2. 用定制化的方法生成 JSON 格式的审计日志。通过需求分析发现，目前对审计日志的格式需求主要还是 JSON，因此我们在内核代码中可以完全消除关于 BSON 的生成、匹配和转换等一系列操作，只需要按照格式生成 JSON 即可。另外，由于审计日志的格式很固定，因此我们也没有使用任何开源的 JSON 库，而是直接使用了字符串拼接的方式，进一步提升了生成审计日志的性能。

# 4. 一些常见的高性能日志机制
多并发写日志，需要同时保证高性能和正确性的场景，想必数据库内核开发者都不会陌生：WAL(journal) 日志就是典型的场景。       
下面结合主流的开源数据库，看看 PostgreSQL/MongoDB(WiredTiger)/MySQL(InnoDB) 都是如何做的。

## 4.1 PostgreSQL 中的 WAL 日志
PostgreSQL 中写 WAL 日志的流程包括用户进程（写数据、提交事务等）以及后台进程（比如 checkpoint, vacuum 进程）。      
为了提升性能，写 WAL 时会先写入到共享内存中，而不是直接写入磁盘上的 WAL 文件。      
后台进程以及用户进程会在合适的时机将共享内存中的 WAL 日志数据刷到磁盘上。    

整体架构如下：

写入流程为：    
1. 先获取插入锁，系统中维护了 8 个插入锁 InsertLocks[8], 每个插入进程获取任意一个插入锁即可。也因为如此，系统的 WAL 插入并发为 8;    
2. 先获取一个 spinLock, 根据本次要插入的日志（XLogRecord）对齐后的长度，得到对应的 SN，并通过线性映射关系得到 LSN（映射的时候需要考虑每个页面的头部大小）。 这个过程非常快，因此直接在 spinLock 的保护下完成；     
3. 完成 XlogRecord 的内存拷贝动作，拷贝的过程中还会按照页面级别更新自己的拷贝进度；    
4. 可能的流程：如果拷贝到的共享内存页面还没有初始化，需要先刷盘（会获取 WALWriteLock）并初始化(会获取 WALBufMapping lock)，初始化完成后才能进行拷贝；    

刷盘流程为（后台进程或者用户进程自己）：    
1. 遍历 InsertLocks[8], 获取各个插入进程的完成情况，确认无空洞（可能会有等待）；
2. 获取 WALWriteLock，确保同一时刻只有一个进程写 WAL 日志文件；
3. 可能的流程：获取 WALBufMapping lock，尽可能将已经刷盘的内存页面初始化，这样尽量降低插入进程的页面初始化开销；     

>注：前面描述的 InsertLocks、WALWriteLock、WALBufMapping lock 是 PostgreSQL 自己实现的具有 EXCLUSIVE/SHARED 2 种模式的锁，内部采用信号量进行等待和唤醒。其中 InsertLocks 除了锁本身的能力之外，还封装了记录和获取插入进度的功能。

PostgreSQL 通过精心设计的锁层次、原子变量、后台进程协作，在保证 WAL 日志正确性和持久性的前提下，最大化了并发插入吞吐量，最小化了事务提交延迟。核心设计思想是：**将耗时的 I/O 操作从关键路径中移出，通过后台进程异步完成；只在绝对必要时（如事务同步提交）才在后端进程中执行刷盘，并利用 Group Commit 分摊 fsync 开销。**
