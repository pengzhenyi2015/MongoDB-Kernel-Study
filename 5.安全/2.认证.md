# 1. 导语      
MongoDB 使用 SCRAM (Salted Challenge Response Authentication Mechanism) 作为主要认证机制，基于 RFC 5802 标准。       
当用户在 MongoDB 8.0 中创建一个用户时，默认会生成 `SCRAM-SHA-1` 和 `SCRAM-SHA-256` 认证的相关信息。      
本文结合 MongoDB 8.0.9 内核代码，以及 mongo-go-driver 库，详细介绍了 SCRAM 认证的实现原理。      

# 2. SCRAM 认证的实现原理
## 2.1 认证之前
连接 MongoDB 之后，使用如下 createUser 命令创建用户：
```
db.createUser({user:"test_user", pwd:"test_pwd", roles:[{role:"root", db:"admin"}]})
```

在 `admin.system.users` 表中，会存储用于认证的如下信息：
```
{
        "_id" : "admin.test_user",
        "userId" : UUID("1624b15c-5910-44dc-98a5-1b01fea59daf"),
        "user" : "test_user",
        "db" : "admin",
        "credentials" : {
                "SCRAM-SHA-1" : {
                        "iterationCount" : 10000,
                        "salt" : "p1UmyXRusJKTYJfnI2nMEA==",
                        "storedKey" : "YZlfoogVtxvU95f2m71trITTWuE=",
                        "serverKey" : "cXv7yTxNAbONApPp+JIq+n0z97E="
                },
                "SCRAM-SHA-256" : {
                        "iterationCount" : 15000,
                        "salt" : "hr1Vj6UVuHHD81wYc3Ues+GCslht4nhw083fDw==",
                        "storedKey" : "Nu3VwvNXC80vofLngMgpOVY6DJv2kjBsCyEEkgtc1uk=",
                        "serverKey" : "etRTUHH8PK1ExMEZwsuToaLhRVqTb+ZmHUk2K4Kb/OA="
                }
        },
        "roles" : [
                {
                        "role" : "root",
                        "db" : "admin"
                }
        ]
}
```
可以看到， 在服务端的 `admin.system.users` 表中，**不会存储任何明文密码！**     
而是存储 `iterationCount`、`salt`、`storedKey` 和 `serverKey` 信息，下面对这 4 个字段的来源和作用进行解释：
|字段|来源|作用|
|--|--|--|
|iterationCount|迭代次数|用于生成 `storedKey` 的迭代次数，SCRAM-SHA-1 的默认值是 1 万，而 SCRAM-SHA-256 的默认值是 1万5千。<br>迭代次数越多，生成 `storedKey` 时执行的 Hash 次数就越多，暴力破解的成本就越大，当然认证时消耗的 CPU 资源也越多|
|salt|盐值|使用安全随机数算法生成的随机数，结合 iterationCount 可以生成加盐的密码：<br>saltedPassword = Hi(password, salt, iterationCount);<br>具体参考 generateSaltedPassword 流程|
|storedKey|（客户端）密码摘要|用于验证用户输入的密码是否正确，计算方式为：<br> ClientKey = HMAC(saltedPassword, "Client Key");<br>StoredKey = SHA(ClientKey);<br>具体参考 Secrets 计算流程|
|serverKey|（服务器）密码摘要|用于生成摘要信息传给客户端，让客户端相信自己知道准确密码。<br>计算方式为：<br>ServerKey = HMAC(saltedPassword, "Server Key");<br> 具体参考 Secrets 计算流程|

由此可见，在执行认证之前：
1. **客户端知道 username、db、password 信息**；
2. **服务端知道 username、db、storedKey、serverKey 信息**。而 storedKey 和 serverKey 都是根据 password、 salt、iterationCount、"Client Key"、"Server Key" 通过哈希计算得到，**并不会存储明文的 password 信息**。

## 2.2 认证过程
认证的目的，无非就是：

- 让服务端知道客户端知道密码；
- 让客户端知道服务端知道密码（双向验证）；

为了实现这个目的，第一想法就是将 salt、iterationCount、storedKey 这些信息告诉客户端，让客户端使用同样的哈希算法计算一遍，然后对比计算结果是否一致？     
但是需要考虑一个问题：如果有中间人截获了这些信息，是否能够执行重发攻击，甚至暴力破解原始密码？    
SCRAM 算法采用了下述策略加以解决：
1. **引入随机数**（客户端和服务端都会产生随机数）；
2. **传递摘要，而不是明文的 storedKey 和 serverKey**（虽然这 2 个也不是原始密码,但是确实是根据原始密码稳定推理出来的摘要，还是需要保护的）；

最终认证过程拆分成了 2 个请求：
- **saslStart**：用于双方传递各自随机数和 salt、iterationCount；
- **saslContinue**：用于双方传递各自的签名，各自验证对方是否知道密码；

### saslStart 请求
Client 构建 saslStart 请求，请求中包含的信息示例如下：
```
{
  "saslStart": 1,
  "mechanism": "SCRAM-SHA-1",
  "payload": {
    "$binary": {
      "base64": "biwsbj11c2VyLHI9ZnlrbytkMmxiYkZnT05Sdjlxa3hkYXdM",
      "subType": "00"
    }
  },
  "options": {
    "skipEmptyExchange": true
  }
}
```
payload 解码后内容：
```
n,,n=user,r=fyko+d2lbbFgONRv9qkxdawL
```
字段说明：
- n, : 无授权绑定
- n=username : 用户名
- r=clientNonce : 客户端生成的随机数

具体流程可以参考 mongo-go-driver 中的 saslConversation::FirstMessage.

### saslStart 响应
响应中包含了 serverNonce（服务端随机数） 、salt、iterationCount 信息。示例如下：
```
{
  "ok": 1,
  "conversationId": 1,
  "done": false,
  "payload": {
    "$binary": {
      "base64": "cj1meWtvK2QybGJiRmdPTlJ2OXFreGRhd0xIytWZ2s3cXZVT0tXd1dMSVc0bCk5U3Jh
      R01IRXg9cy1rUTlaTTNudEJldVAzRTFUREtDNHc9P0k9MTAwMDA=",
      "subType": "00"
    }
  }
}
```
payload 解码后内容：
```
r=fyko+d2lbbFgONRv9qkxdawLHo+Vgk7qvUOKUwuWLIWg4l/9SraGMHEE,s=rQ9ZY3MntBeuP3E1TDVC4w==,i=10000
```
字段说明：
- r=serverNonce : 服务器随机数（客户端随机 + 服务器随机拼接）
- s=salt : Base64 编码的盐值
- i=iterations : 迭代次数

详细流程参考 [SaslSCRAMServerMechanism\<Policy\>::_firstStep]().

### saslContinue 请求
客户端通过这个请求携带自己的证明，请求中包含的数据示例如下：
```
{
  "saslContinue": 1,
  "conversationId": 1,
  "payload": {
    "$binary": {
      "base64": "Yz1iaXdtbmNvPWZ5a28rZDJsYmJGZ09OUnY5cWt4ZGF3TEhvK1ZnazdxdlVPS1d1d1d
      MSVc0bCk5U3JhR01IRXgscD1DNGpEMytxWEpkNWtKQU5hdzZXdzVHcUhKSkFPQ==",
      "subType": "00"
    }
  }
}
```
payload 解码后内容：
```
c=biws,r=fyko+d2lbbFgONRv9qkxdawLHo+Vgk7qvUOKUwuWLIWg4l/9SraGMHEE,p=CLjD2+qXJd5kJANaw2WwTWwRIJAO
```
字段说明：

- c=biws : 绑定信息，base64 编码的 n,,
- r=full-nonce : 客户端+服务器组合随机数
- p=clientProof : 客户端证明，计算方式：
  1. saltedPassword = PBKDF2(hashedPassword, salt, iterations, SHA1)
  2. clientKey = HMAC(saltedPassword, "Client Key")
  3. storedKey = SHA1(clientKey)
  4. authMessage = client-first-bare + "," + server-first + "," + client-final-without-proof
  5. clientSignature = HMAC(storedKey, authMessage)
  6. clientProof = clientKey XOR clientSignature ，然后 base64 编码

>authMessage 会将目前为止所有交互的信息组合到一起，用于生成证明，这种方式进一步降低重放攻击的可能性。
计算方式为：
client-first-bare = `n=username,r=client-nonce` ← 客户端第一条消息（去掉了 n, 前缀）
server-first = `r=full-nonce,s=salt,i=iterations`← 服务端第一条消息
client-final-without-proof = `c=bind,r=full-nonce`← 客户端最后消息（去掉了 p=proof 部分）
authMessage = `client-first-bare + "," + server-first + "," + client-final-without-proof`


具体流程可以参考 mongo-go-driver 中的 saslConversation::Finish.

### saslContinue 响应
服务端检验客户端证明是否正确，计算方法：
1. 根据收到的 ClientProof 、本地缓存的 AuthMessage 和存储的 StoredKey，反解 ClientKey = HMAC(StoredKey, AuthMessage) XOR ClientProof
2. 计算 exp = SHA(ClientKey) （期望的 StoredKey ）
3. 常量时间比较 exp 和存储在数据库中的 StoredKey 是否相等

如果 ClientProof 正确，返回成功响应，并携带自己的证明。响应中包含的数据示例如下：
```
{
  "ok": 1,
  "conversationId": 1,
  "done": true,
  "payload": {
    "$binary": {
      "base64": "dj1VTVdlSTI1SkQxeU5ZWlJNcCo0Vkh2aFo5ZTA9",
      "subType": "00"
    }
  }
}
```
payload 解码后内容：
```
v=UMWeI25JD1yNYZRMpZ4VHvhZ9e0=
```

字段说明：
- v=serverSignature : 服务器签名，验证客户端证明正确后发送，用于客户端验证服务器

生成 serverSignature 的逻辑参考 Secrets::generateServerSignature.

客户端收到响应之后，会进行 serverSignature 的验证，确保服务端也知道密码。    
验证流程如下(参考 xdg-go 3方库中的 ClientConversation::finalMsg 和 ClientConversation::validateServer 方法)：
1. 客户端计算 serverKey = HMAC(saltedPassword, "Server Key")
2. 客户端计算 expectedServerSignature = HMAC(serverKey, authMessage)
3. 客户端比对接收到的 v 是否与计算结果一致
4. 如果一致，认证成功完成

# 3. 总结
SCRAM 认证流程具有如下安全特征：
1. 不传输明文密码
- 服务端只存储 StoredKey 和 ServerKey
- 客户端证明通过 HMAC 和 XOR 计算，不暴露密码
2. 双向认证
- 客户端通过 ClientProof 证明拥有密码
- 服务端通过 ServerSignature 证明拥有 ServerKey
3. 防重放攻击
- 使用随机 nonce 确保每次认证唯一
- AuthMessage 包含完整的交互历史
4. 防时序攻击
- 使用常量时间比较函数 consttimeMemEqual，防止攻击者根据返回之间的长短逐步推测密码
5. 密钥派生 (PBKDF2)
- 高迭代次数 (最小 4096) 增加暴力破解成本
- 每个用户使用独立的 salt
