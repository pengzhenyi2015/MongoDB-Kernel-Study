# 1. 导语
如何给不同用户分配不同的权限，保证其在受限制的情况下执行操作，是数据库安全的基础。MongoDB 采用基于角色的访问控制（RBAC, Role Based Access Control）来实现此功能。      
本章节结合 8.0.9 内核代码，围绕下面几个关键问题进行阐述：    
1. 如何定义用户权限，user, role 和 privilege 分别代表哪些含义，以及如何存储这些信息？     
2. 如何判断用户是否有权限执行某个命令？
3. 如何通过 UserCache 优化性能？

# 2. 权限的组成

MongoDB 的权限体系，自底向上依次为 privilege, role 和 user.    

## 2.1 Privilege
Privilege 可以理解为：在某些 resource 上可以执行哪些 action.    

__[resource](https://www.mongodb.com/docs/manual/reference/resource-document/)__

resource 定义了具体的资源：常见的有 cluster（集群资源）, db（库）, collection（表）,以及 anyResource（任何资源） 和 system_buckets（时序集合）。 

比如对整个集群的操作：
```
{resource: {cluster: true}}
```
对某个 db 的操作：
```
{resource: {"db":"testdb", collection: ""}}
```
对某个表的操作：
```
{resource: {"db":"testdb", "collection":"testcollection"}}
```

__[action](https://www.mongodb.com/docs/manual/reference/privilege-actions/)__

参考[官方文档](https://www.mongodb.com/docs/manual/reference/privilege-actions/)中的定义，内置的 action 分为 读写操作、集群管理、诊断操作、changeStream、分片操作、内部操作等。       
所有内置的 action 定义在 [action_type.idl](https://github.com/mongodb/mongo/blob/r8.0.9/src/mongo/db/auth/action_type.idl) 文件中，并且规定了 action 和 resource 的匹配规则。    

比如定义对某个表的只读权限：
```
{resource: {db: "db1", collection: "coll1"}, actions:["find"]}
```

定义对集群的 serverStatus 管理命令的权限和 listDatabases 执行权限：
```
{resource: {cluster: true}, actions: ["serverStatus", "listDatabases"]}
```

## 2.2 Role

系统自定义的 role 可以参考[官方文档](https://www.mongodb.com/docs/manual/reference/built-in-roles/?deployment-type=self)说明。     

在代码中的定义参考 [builtin_roles.yml](https://github.com/mongodb/mongo/blob/r8.0.9/src/mongo/db/auth/builtin_roles.yml), 其中定义了每个内置 role 包含的 privileges, 以 readAnyDatabase 为例，定义如下：
```
readAnyDatabase:
    adminOnly: true                       # 只应用于 admin 库
    privileges:
        - matchType: any_normal           # 对普通库和普通集合
            actions: *readRoleActions     # readRoleActions 定义在前面的代码中，包含了一组 action 集合，这里只是引用
        - matchType: cluster              # 对整体集群的权限
            actions:
                - listDatabases
        - matchType: collection           # 对特殊表的权限
            collection: 'system.js'
            actions: *readRoleActions
        - matchType: any_system_buckets   # 对时序表的权限
            actions: *readRoleActions

# readRoleActions 定义在代码文件的前面，包含的 privileges 如下
actions: &readRoleActions
    - changeStream
    - collStats
    - dbHash
    - dbStats
    - find
    - killCursors
    - listCollections
    - listIndexes
    - listSearchIndexes
    - planCacheRead
```
另外， role 之间可以有继承关系，比如 [dbOwner](https://github.com/mongodb/mongo/blob/r8.0.9/src/mongo/db/auth/builtin_roles.yml#L132) 继承自 readWrite, dbAdmin, userAdmin:
```
dbOwner:
    roles: [ readWrite, dbAdmin, userAdmin ]
```

除了内置的 role 之外，用户可以通过 createRole 命令创建自定义 role，创建的时候可以指定 privilege 以及继承的 role 列表：  
```
db.createRole({role:"testAdmin",
               privileges:[
                {resource: {cluster: true},
                actions: ["serverStatus", "listDatabases"]}
               ],
               roles:[]})
```
role 创建成功后，会持久化到 `admin.system.roles` 系统表中：
```
{
    "_id" : "admin.testAdmin",
    "role" : "testAdmin",
    "db" : "admin",
    "privileges" : [
        {
            "resource" : {
                    "cluster" : true
            },
            "actions" : [
                    "listDatabases",
                    "serverStatus"
            ]
        }
    ],
    "roles" : [ ]
}
```

## 2.3 User
创建一个用户一般指定的选项有：
- user: 用户名；
- pwd: 密码；
- roles: 角色数组，每项角色包括 role (角色名) 和 db (创建角色时 use 的 db)；
- authenticationRestrictions: 可以对认证的 clientSource 和 serverAddress 进行限制，在前面创建 role 时也能指定此类限制；
- mechanisms: 认证的算法；

一个 user 可能包含很多 roles，不同的 role 之间可能存在 privilege 重叠或者继承关系。    
在内核代码中，使用 User 类包含了一个具体的 user 的属性。在内存中初始化 User 对象时，会将包含的 roles 权限进行合并，并平铺成一个 hash 表进行存储，方便快速查找。    

具体来说：
|**设计点**|**实现方式**|**时间复杂度**|
|:--|:--|:--|
|权限查找|unordered_map<ResourcePattern, Privilege>|**O(1)** 平均|
|角色判断|unordered_set<RoleName>|**O(1)** 平均|
|Action 存储|std::bitset<kNumActionTypes>|**O(1)** 位运算|

# 3. 操作权限的验证逻辑
## 3.1 普通操作
在实际的使用和开发中，我们常遇见的一个问题是：对于某个操作，如何知道需要分配哪些最小化的 role/privilege？    

对于一些常见的操作，是显而易见的，比如执行 find 命令需要有 find 权限，执行 replSetGetStatus 管理命令需要有 replSetGetStatus 权限等。    
除此之外，在内核代码中也会有一些特殊处理。比如即使没有 listCollections 权限，但是对某些库表有 find 等权限时，也能执行 listCollections 操作，只是仅返回那些有权限的表。    

关于权限的检查，可以参考每个命令具体的 doCheckAuthorization 方法。

## 3.2 grant/revoke 权限操作
和前面描述的类似，执行 grant/revoke 操作时，执行操作的用户需要有对应 db 的 grantRole/revokeRole 的 action 权限。

而这些权限一般属于下面 3 个内置 role:    
|**角色**|**grantRole**|**revokeRole**|**作用范围**|
|:--|:--|:--|:--|
|userAdmin/dbOwner|√|√|单个数据库|
|userAdminAnyDatabase|√|√|所有数据库|
|root/restore|√|√|整个集群|
